## 生成流程讲解

**（指向右侧生成日志）**

生成过程中，右边会实时显示生成日志。趁这个时间我讲一下生成流程。

### 节拍提取

我们首先对整个小说提取"故事节拍"。节拍是什么？节拍就是最小的剧情单元——一个人做了什么事，或者一个事件发生了，就是一个节拍。

为什么要提取节拍？因为小说原文里很多是抒情、铺垫之类的文字，这些不利于 AI 抓住故事的核心。我们把小说提炼成节拍，就等于把小说处理成了一个核心骨架。

有了节拍之后，我们把所有节拍交给 AI，让 AI 通过每个节拍的 meta-data 去分析——人物是谁、场景在哪里、冲突是什么——然后把相关的节拍合并成一个场景。一般是多个节拍对应一个剧本场景。
### 场景生成
节拍确定了之后，AI 就开始针对每个场景生成剧本内容——把节拍展开成具体的动作和对白。

### 场景计划

生成场景之后，系统会对每个场景做详细计划——这个场景里要发生什么、每个人要做什么动作、说什么话。

### 剧本合并

场景是并行生成的，目的是提高效率。所以生成完之后需要合并，把每个场景里的人物和场景汇聚到一起。

### 总编审查

合并之后，总编 Agent 做一次统一审查。审查通过，就直接展示剧本。

### 查看生成过程

**（点开 Jaeger / Trace）**

我们接了一个 Jaeger 做可观测。可以看到整个生成过程中每个步骤的耗时——节拍提取、场景编排、场景计划、生成每个场景。

## 导出

**（点击导出）**

最后需要导出成 YAML 文件。导出在这里。除了 YAML，我们也支持其他格式。

**（展示 YAML 结构）**

讲一下 YAML 的格式。所有内容包在 screenplay 这个根节点下面。metadata 是基本信息，然后是角色列表，然后是场景列表。

每个场景有它自己的 ID、序号、场景标题、设定——比如位置、时间——然后是大致的剧情流程、出场人物。内容部分具体就是 action、dialogue——是动作还是对白。

为什么这么设计？核心元素就是 action 和 dialogue。我依据这个核心，再补充了相关的 metadata——这就是我们这个格式的设计缘由。
