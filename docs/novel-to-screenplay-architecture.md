# AI 小说转剧本工具整体架构设计

## 1. 项目目标

本项目要解决的问题是：小说文本通常很长，人物、事件、伏笔和关系变化复杂，直接把全文交给 AI 生成剧本容易遗漏、混乱或泄露后文信息。因此系统需要先理解小说结构，再控制每次生成剧本时 AI 能看到的上下文。

最终目标是：

- 支持非常长小说输入；
- 自动将长篇小说拆解成可处理的剧情单元；
- 维护随剧情推进变化的人物关系与故事状态；
- 基于叶子剧情单元生成结构化剧本；
- 输出可编辑、可校验的 YAML 剧本初稿；
- 保持最终 YAML Schema 简洁，不把内部复杂分析结构暴露给作者。

## 2. 核心设计思想

整体方案采用“递归故事树 + 动态人物关系图 + 场景上下文包”的混合架构。

### 2.1 递归故事树

小说先被看作一棵故事树。

- 根节点代表整部小说；
- 第一层节点来自硬切割后的文本块；
- 中间节点代表仍然过长或过复杂的剧情段落；
- 叶子节点代表语义完整、长度可控、可以进入剧本改编流程的剧情单元。

递归故事树的作用不是生成剧本，而是解决长文本理解问题：把大文本逐步拆成 AI 能稳定理解的小单元。

### 2.2 横向连续链

同一层的节点按原文顺序连接成链。

这样做是为了处理硬切割造成的截断问题。例如一段对话、一次动作或一个事件刚好被切到两个节点里，系统可以让当前节点向右查看相邻节点，并在确认属于同一未完成事件时合并。

横向链只用于修复截断，不用于无限扩展上下文。

### 2.3 动态人物关系图

人物关系不是一次性抽取出的静态结果，而是随着剧情推进不断更新的动态图。

每处理一个叶子节点，系统会更新：

- 出场人物；
- 人物当前目标；
- 人物情绪状态；
- 人物已知信息；
- 人物之间的关系；
- 关系变化的触发事件；
- 尚未解决的冲突和伏笔。

动态图的关键作用是：让每个场景只看到当前时间点之前已经发生的信息，避免后文真相提前泄露。

### 2.4 场景上下文包

叶子节点不会直接裸交给剧本写作 Agent。

系统会先根据叶子节点生成场景计划，再结合动态图组装一个上下文包。上下文包只包含当前场景需要的信息，例如：

- 当前叶子节点原文或摘要；
- 当前场景的戏剧目标；
- 出场人物；
- 出场人物之间当前可见的关系；
- 当前人物已知信息；
- 必须保留的剧情点；
- 可以省略的小说细节；
- 禁止使用的后文信息边界。

默认策略是：程序根据出场人物自动从动态图中取相关关系和事件，直接放进上下文包。必要时可以扩展为受限查询，但查询必须限制在当前节点之前的故事状态。

## 3. 整体工作流

### Step 1：输入与预处理

用户输入至少 3 个章节的小说文本。

系统进行基础预处理：

- 保留章节顺序；
- 清理无效空白；
- 记录文本来源位置；
- 保留章节编号；
- 为后续节点追溯建立基础索引。

输出结果是可追溯的原始文本集合。

### Step 2：硬切割生成第一层故事节点

系统先按固定规则切割文本，优先考虑：

- 章节边界；
- 自然段落；
- 最大上下文长度；
- 句子或段落完整性。

这一阶段不依赖 AI 深度理解，主要目的是保证每个文本块长度可控。

输出结果是第一层故事节点。

### Step 3：建立横向连续链

同层故事节点按原文顺序连接。

例如：

```text
node_001 -> node_002 -> node_003 -> node_004
```

每个节点知道自己的左侧节点和右侧节点。

这条链用于后续判断：当前节点是否被硬切割截断，是否需要向右获取相邻内容。

### Step 4：AI 分析故事节点

节点分析 Agent 读取单个节点，并输出结构化理解结果。

每个节点至少需要判断：

- 这段剧情讲了什么；
- 主冲突是什么；
- 出场人物有哪些；
- 发生了哪些事件；
- 地点和时间是否变化；
- 人物关系是否发生变化；
- 当前节点是否语义完整；
- 是否需要继续拆分；
- 是否需要向右合并；
- 是否适合改编为剧本场景。

节点分析后会给出处理决策。

### Step 5：节点处理决策

每个节点只能进入以下几种处理路径：

#### keep

节点剧情完整、长度可控，可以成为叶子节点。

#### split

节点仍然太长或太复杂，需要继续向下拆分。

常见原因：

- 多个事件混杂；
- 多个地点切换；
- 时间跨度过大；
- 出场人物过多；
- 主线和支线混在一起；
- AI 无法用一句话概括主冲突。

#### merge_right

当前节点被截断，需要查看右侧相邻节点。

只在以下情况合并：

- 对话被切断；
- 动作正在进行；
- 当前事件结果缺失；
- 不看右侧就无法理解当前事件。

不因为“后文还有影响”而合并。

#### summarize_only

节点有信息价值，但不适合直接改编成戏。

例如：

- 背景说明；
- 世界观解释；
- 长段心理活动；
- 过场交代。

这类内容进入摘要或人物状态，不直接生成场景。

#### discard

节点对剧本初稿没有明显价值，也不影响人物关系和主线。

这个动作应谨慎使用，默认更倾向于 `summarize_only`。

### Step 6：递归拆分直到得到叶子节点

如果节点被判断为 `split`，系统继续把它拆成子节点，并重复分析流程。

递归停止条件：

- 节点长度安全；
- 事件相对完整；
- 主冲突清楚；
- 人物状态清楚；
- 可以判断是否进入场景生成；
- 不再需要继续拆分。

最终得到一组按原文顺序排列的叶子节点。

### Step 7：按叶子节点顺序更新动态图

系统按故事顺序处理叶子节点。

对于每个叶子节点：

1. 读取当前节点之前的图状态；
2. 分析当前节点中的事件、人物和关系变化；
3. 更新人物目标、情绪、已知信息；
4. 更新人物关系；
5. 记录触发关系变化的事件；
6. 生成当前节点之后的新图状态。

因此每个叶子节点都有两个关键状态：

- `state_before_node`：当前节点之前的故事状态；
- `state_after_node`：当前节点之后的故事状态。

剧本生成只能使用 `state_before_node` 加当前节点内部可见信息，不能使用后续节点的状态。

### Step 8：叶子节点生成场景计划

叶子节点不直接等于剧本场景。

系统先判断每个叶子节点应该如何进入剧本：

- 生成 0 场戏：背景、说明、心理活动；
- 生成 1 场戏：单地点、单时间、单冲突；
- 生成多场戏：地点、时间或冲突阶段发生明显变化；
- 与相邻叶子节点合并成一场戏：短连续动作、同一地点、同一目标。

场景计划需要说明：

- 场景目的；
- 来源叶子节点；
- 地点；
- 时间；
- 出场人物；
- 必须保留的剧情点；
- 可以省略的小说细节；
- 该场景在全剧中的顺序。

### Step 9：组装场景上下文包

每个场景生成前，系统根据场景计划和动态图组装上下文包。

上下文包包括：

- 场景计划；
- 来源叶子节点内容；
- 相关原文或摘要；
- 当前出场人物信息；
- 出场人物之间当前可见关系；
- 当前人物已知信息；
- 与本场有关的前置事件；
- 尚未解决但与本场有关的冲突；
- 需要避免提前揭示的信息；
- YAML Schema 输出要求。

这一步是整个系统控制 AI 输出质量的关键。

### Step 10：场景写作 Agent 生成 YAML 场景

场景写作 Agent 只处理一个场景上下文包。

它的任务是：

- 将小说叙述转换为可见动作；
- 将适合对白化的内容转换为对白；
- 保留必要冲突；
- 删除或压缩不适合剧本表现的说明；
- 输出单个结构化 YAML 场景。

场景写作可以并行执行，因为每个场景都有独立上下文包。

但并行写作不能改变故事顺序，最终合并仍按场景计划排序。

### Step 11：合并场景

系统将所有单场景 YAML 按场景顺序合并为完整剧本。

完整剧本包含：

- `metadata`；
- `characters`；
- `scenes`。

内部故事树、动态图、场景计划和上下文包不进入最终 YAML。

### Step 12：总编 Agent 审查

总编 Agent 对完整剧本进行统一审查。

主要检查：

- 剧情顺序是否连贯；
- 人物口吻是否一致；
- 人物关系是否前后矛盾；
- 是否泄露后文信息；
- 场景节奏是否过散；
- 同一场景内动作和对白是否服务冲突；
- YAML 结构是否稳定。

总编 Agent 只能基于已有故事状态修正，不能新增无来源剧情。

### Step 13：程序校验

最终输出前，系统做程序级校验。

校验内容包括：

- YAML 是否能解析；
- 必需字段是否存在；
- 场景顺序是否连续；
- 对白角色是否存在；
- `characters_present` 是否引用合法角色 ID；
- 场景内容是否为空；
- 是否至少覆盖 3 个来源章节。

校验失败时，系统可以进入一次修复流程；如果仍失败，则返回错误和原始输出，方便人工排查。

### Step 14：输出最终 YAML 剧本

最终输出的是作者可编辑的 YAML 剧本初稿。

输出只保留作者需要修改和使用的内容，不暴露内部分析结构。

## 4. 动态图如何提供给 AI

这里采用“默认注入 + 受限查询”的组合策略。

### 4.1 默认注入

系统根据当前场景的出场人物，自动从动态图中取出相关信息，放入上下文包。

例如当前场景出场人物是林浩和苏晴，系统会提供：

- 林浩当前目标；
- 苏晴当前目标；
- 两人当前关系；
- 两人此前共同经历的关键事件；
- 两人分别知道和不知道的信息；
- 与当前场景冲突相关的未解决问题。

这是主路径。

优点：

- 稳定；
- 可控；
- 不容易超上下文；
- 不容易泄露未来信息；
- 实现成本较低。

### 4.2 受限查询

如果后续实现 Agent 希望让写作 Agent 自行查询，可以提供受限查询能力。

查询必须遵守：

- 只能查当前节点之前的信息；
- 查询必须带当前节点边界；
- 不能访问完整全局图；
- 不能查询未来节点；
- 查询结果要尽量短，只返回与当前场景有关的信息。

可以支持的查询类型：

- 查询某个人物当前状态；
- 查询几个出场人物之间的关系；
- 查询与出场人物有关的前置事件；
- 查询当前仍未解决的冲突；
- 查询某个伏笔是否已经在当前时间点出现。

推荐先实现默认注入，受限查询作为增强能力。

## 5. Agent 分工

### 5.1 流程调度器

负责控制整体流程。

它不直接创作剧本，而是负责任务编排：

- 切分；
- 递归；
- 合并；
- 图更新；
- 上下文包组装；
- 场景生成调度；
- 最终校验。

### 5.2 节点分析 Agent

负责理解故事节点。

输出内容包括：

- 摘要；
- 主冲突；
- 事件；
- 人物；
- 地点时间；
- 关系变化；
- 是否完整；
- 是否继续拆分；
- 是否适合改编。

### 5.3 关系状态 Agent 或程序模块

负责维护动态图。

可以由程序主导，AI 辅助判断关系变化。

关键要求：

- 必须按故事顺序更新；
- 必须记录变化来源；
- 必须区分人物真实身份和当前可见认知；
- 必须防止未来信息进入过去场景。

### 5.4 场景规划 Agent

负责把叶子节点转成场景计划。

它决定：

- 哪些叶子节点生成场景；
- 一个叶子节点生成几场；
- 哪些叶子节点只保留摘要；
- 哪些短节点可以合并；
- 每场戏的戏剧目的是什么。

### 5.5 场景写作 Agent

负责根据上下文包生成单个 YAML 场景。

它只关心当前场景，不负责全局结构。

### 5.6 总编 Agent

负责最终统一剧本。

它检查连贯性、人物一致性、风格统一和格式问题。

### 5.7 程序校验器

负责确定最终结果是否合格。

它不判断文学质量，只判断结构是否可靠。

## 6. 需要特别注意的点

### 6.1 不要让图谱复杂度超过项目目标

动态图是辅助上下文控制的工具，不是项目主体。

不要为了构建复杂图谱而牺牲剧本生成效果。

重点始终是：

- 剧情不乱；
- 人物不乱；
- 上下文不过量；
- YAML 可编辑。

### 6.2 不要把未来信息泄露给过去场景

这是整个系统最重要的约束之一。

尤其是：

- 反派身份；
- 隐藏关系；
- 后文揭示的动机；
- 尚未发生的背叛；
- 尚未回收的伏笔。

这些信息不能出现在早期场景的上下文包中。

### 6.3 叶子节点不等于场景

叶子节点是小说理解单位，场景是剧本表达单位。

两者需要通过场景计划连接，不能机械地一一对应。

### 6.4 合并只解决截断，不解决长线剧情

如果当前事件被切断，可以向右合并。

但如果只是同一主线后面还有发展，不应该合并，否则会重新制造长上下文问题。

### 6.5 AI 负责理解与创作，程序负责边界与校验

AI 适合做：

- 语义理解；
- 冲突判断；
- 人物状态总结；
- 场景创作；
- 语言润色。

程序适合做：

- 切分；
- 排序；
- 限制上下文；
- 图状态版本控制；
- YAML 解析；
- 字段校验；
- 角色引用校验。

不要把边界控制完全交给 AI。

### 6.6 最终 YAML 要保持简洁

内部可以有故事树、动态图、上下文包和场景计划。

但最终交给作者的 YAML 应该只包含：

- 基本信息；
- 角色表；
- 场景列表；
- 场景内容。

作者不应该被内部工程结构打扰。

### 6.7 场景生成应该尽量局部化

每次只生成一个场景或一组强相关场景。

不要让一个 Agent 一次性生成整部剧本。

这样可以降低遗漏、跑题和格式错误的概率。

### 6.8 并行生成之后必须有统一审查

场景写作可以并行，但并行会带来风格不一致和细节冲突。

因此必须有总编阶段统一处理。

### 6.9 对失败要有可恢复路径

常见失败包括：

- 节点分析 JSON 解析失败；
- YAML 缩进错误；
- 角色引用不存在；
- 场景顺序错乱；
- 上下文包过长；
- 生成内容泄露后文信息。

系统应尽量返回具体失败位置，而不是只返回“生成失败”。

## 7. 推荐落地顺序

推荐按以下顺序实现：

1. 先实现故事树切分和叶子节点生成；
2. 再实现节点分析结构；
3. 再实现动态图状态更新；
4. 再实现叶子节点到场景计划；
5. 再实现上下文包组装；
6. 再实现单场景 YAML 生成；
7. 再实现总编修订；
8. 最后实现最终校验和错误恢复。

不要一开始就做完整 Agent 自动化。先把流程边界跑通，再逐步增强每个 Agent 的能力。

## 8. 一句话总结

这个系统的核心不是让 AI 一次读完整本小说，而是先把小说变成可控的故事树，再用动态图维护人物和剧情状态，最后让 AI 在精准上下文中逐场生成 YAML 剧本。
